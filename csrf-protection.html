Referer Header Check

Effectiveness:

Checking the HTTP Referer header to ensure requests originate from your own domain.

‚úÖ Advantages

  • Simple to implement
  • No client-side changes needed
  • Works for all request types

‚ùå Limitations

  • Browsers might strip the header
  • Privacy tools may block it
  • Referrer-Policy can disable it
  • Potentially spoofable

Implementation Example:

// Express.js implementation app.post('/transfer', (req, res) => { const referer = req.headers.referer; // Check if Referer exists and is from our domain if (!referer || !referer.startsWith('https://bank.com')) { res.status(403).send('Invalid referer'); return; } // Process transfer... res.send('Transfer successful'); });
‚ö†Ô∏è Warning: Never rely solely on Referer header checks for CSRF protection. Use this only as an additional layer.

SameSite Cookie Attribute

Effectiveness:

Using the SameSite cookie attribute to control when cookies are sent with cross-site requests.

SameSite=Lax (Recommended Default)

Cookies are sent with top-level navigations but not with cross-site subrequests.

  • ‚úÖ CSRF protection for state-changing requests
  • ‚úÖ Allows legitimate navigation from external sites
  • ‚úÖ Prevents CSRF via form submissions

SameSite=Strict (Maximum Security)

Cookies are only sent for same-site requests.

  • ‚úÖ Maximum CSRF protection
  • ‚ùå Breaks links from external sites
  • ‚ùå Users must start navigation from your site

SameSite=None (No Protection)

Cookies are always sent, even with cross-site requests.

  • ‚ùå No CSRF protection
  • ‚úÖ Required for cross-site functionality
  • ‚ö†Ô∏è Must use with Secure attribute

Implementation Example:

// Express.js cookie setup res.cookie('sessionId', sessionId, { httpOnly: true, // Prevent JavaScript access secure: true, // Require HTTPS (essential for SameSite) sameSite: 'lax', // CSRF protection level path: '/', maxAge: 24 * 60 * 60 * 1000 // 24 hours });
üìÖ Browser Update: Since 2020, most browsers default to SameSite=Lax for cookies without an explicit SameSite attribute.

CSRF Tokens (Synchronizer Token Pattern)

Effectiveness:

Generating unique, unpredictable tokens that must be included with each state-changing request.

How CSRF Tokens Work:

Step 1: Server Generates Token
const crypto = require('crypto'); const csrfToken = crypto .randomBytes(32) .toString('hex');
Step 2: Token Verification
// Server verifies token if (req.body.csrf !== req.session.csrfToken) { return res.status(403); }

Complete Implementation:

// Generate and store token app.get('/transfer', (req, res) => { const csrfToken = crypto .randomBytes(32) .toString('hex'); req.session.csrfToken = csrfToken; res.send(` <form method="POST" action="/transfer"> <input type="hidden" name="csrf" value="${csrfToken}" /> Amount: <input name="amount" /> To: <input name="to" /> <input type="submit" value="Send" /> </form> `); });

GET vs POST Requests

Effectiveness:

Using proper HTTP methods to reduce CSRF attack surface.

‚úÖ Use POST for:

  • State-changing operations
  • Sensitive actions
  • Form submissions
  • Data modifications

‚ùå Avoid GET for:

  • Transferring funds
  • Changing passwords
  • Deleting data
  • Any action with side effects

Why POST Reduces CSRF Risk:

GET requests can be forged easily:

<!-- Simple image tag attack --> <img src="http://bank.com/transfer?to=attacker&amount=1000">

Proper HTTP Method Usage:

// ‚ùå WRONG - Using GET for state changes app.get('/transfer', (req, res) => { // This is vulnerable to CSRF! transferMoney(req.query.amount, req.query.to); }); // ‚úÖ CORRECT - Using POST for state changes app.post('/transfer', (req, res) => { // Attack surface is reduced transferMoney(req.body.amount, req.body.to); });
Best Practice: Follow RESTful principles: GET for reading, POST for creating, PUT/PATCH for updating, DELETE for deleting.

üöÄ Recommended Security Strategy

For maximum protection against CSRF attacks, implement a defense-in-depth approach:

  1. Primary Defense: Use SameSite=Lax cookies for all session cookies
  2. Critical Actions: Implement CSRF tokens for sensitive operations
  3. HTTP Methods: Use POST for all state-changing operations
  4. Additional Layer: Consider Referer header checks
  5. Always Use HTTPS: Essential for cookie security
// Complete security setup example res.cookie('sessionId', sessionId, { httpOnly: true, // Prevent XSS access secure: true, // HTTPS only sameSite: 'lax', // CSRF protection path: '/', maxAge: 86400000 }); // Validate CSRF tokens for sensitive actions function validateCSRF(req) { return req.body.csrfToken === req.session.csrfToken; }
