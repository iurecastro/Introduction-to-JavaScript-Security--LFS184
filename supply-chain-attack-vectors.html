Supply Chain Attack Vectors

Entry Points for Supply Chain Compromise

Typosquatting Attacks

Attackers register package names similar to popular packages, hoping developers will make typographical errors when installing dependencies.

Real-World Examples:

  • loadash instead of lodash (popular utility library)
  • express-js instead of express (web framework)
  • reactdom instead of react-dom (React package)
  • axios-http instead of axios (HTTP client)
‚ö†Ô∏è AI-Powered Variant: "Slopsquatting" targets AI-hallucinated package names from LLM-generated commands.

How It Works:

  1. Attacker researches popular packages with high download counts
  2. Registers similar-sounding package names in public registries
  3. Packages contain malicious code in install/postinstall scripts
  4. Developer makes typo when running npm install
  5. Malicious package installs and executes attacker's code
# Example malicious package.json with postinstall script { "name": "loadash", "version": "1.0.0", "description": "A utility library (typosquat of lodash)", "scripts": { "postinstall": "node steal-env.js" }, "dependencies": {} }

Dependency Confusion

Attackers exploit how package managers resolve names between private and public registries.

üí• High-Profile Case: Alex Birsan's "How I Hacked Into Apple, Microsoft and Dozens of Other Companies" demonstrated this attack vector.

Attack Scenario:

  1. Company uses internal packages (e.g., @company/logger)
  2. Package name isn't registered on public npm registry
  3. Attacker finds internal package name in public source code
  4. Attacker registers same name on public npm registry
  5. When CI/CD runs npm install, it pulls malicious public package

Technical Details:

# .npmrc configuration issue # Problem: No scope-specific registry configured # This allows dependency confusion attacks # Vulnerable .npmrc: registry=https://registry.npmjs.org/ # Secure .npmrc: @company:registry=https://internal.company.com/registry/ registry=https://registry.npmjs.org/
üîê Prevention: Always configure scope-specific registries for internal packages in .npmrc.

Account Compromise

Attackers gain access to maintainer accounts to publish malicious versions of legitimate libraries.

Common Attack Methods:

  • Phishing: Fake npm/login emails targeting maintainers
  • Credential Theft: Stolen passwords from data breaches
  • CI Credential Compromise: Malicious PRs that steal CI secrets
  • Social Engineering: Impersonating support staff

üìö Case Study: event-stream Incident

In 2018, the popular event-stream package (2M+ weekly downloads) was compromised when the maintainer transferred ownership to an attacker.

The attacker added a malicious dependency (flatmap-stream) that targeted a specific Bitcoin wallet application.

Impact: The backdoor remained undetected for months, affecting thousands of applications.

Protection Measures:

  • Enable 2FA/MFA on all package registry accounts
  • Use API tokens with minimal permissions
  • Regularly audit account activity and access logs
  • Implement automated security alerts for suspicious activity

Maintainer Burnout Exploitation

Attackers target overburdened maintainers, offering to help maintain projects, then pushing malicious code after gaining trust.

‚ö†Ô∏è Critical Vulnerability: The xz-utils backdoor attack (2024) involved an attacker making legitimate contributions for 2 years before inserting a backdoor.

Warning Signs of Burnout:

  • Slow response to issues and PRs
  • Expressions of frustration in communications
  • Long periods of project inactivity
  • Public statements about maintenance challenges

Attack Timeline (xz-utils Example):

  1. 2021: Attacker starts making legitimate contributions
  2. 2022: Gains commit access through consistent helpfulness
  3. 2023: Becomes co-maintainer
  4. 2024: Inserts backdoor in test files
  5. Backdoor discovered just before widespread deployment
üí° Open Source Sustainability: Many maintainers face burnout, harassment, and lack financial support, making projects vulnerable.

Prevention Strategies:

  • Support open source projects financially or through contributions
  • Implement multi-maintainer review processes
  • Use automated security scanning for all contributions
  • Establish clear maintainer succession plans
