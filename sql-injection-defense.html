SQL Injection Defense Guide

Parameterized queries, ORM protection, and best practices

SQL Injection Defense Overview

‚ö†Ô∏è Critical Security Principle: Never build SQL queries using string concatenation with untrusted input. This is the #1 cause of SQL injection vulnerabilities.

Unsafe vs Safe Examples

// UNSAFE - String concatenation
const query = `SELECT * FROM users
WHERE username = '${username}'
AND password = '${password}'`
Vulnerable to:
username: ' OR '1'='1
Result: WHERE username = '' OR '1'='1'
// SAFE - Parameterized query
const query = 'SELECT * FROM users WHERE username = ? AND password = ?'
const results = db.all(query, [username, password])
Protected: Database driver properly escapes special characters

ORM Protection

Object-Relational Mapping libraries automatically handle parameterization:

// Using Sequelize ORM
const users = await User.findAll({
where: {
username: username,
password: password // Never store plain passwords!
}
})
// Using TypeORM
const user = await userRepository.findOne({
where: { username: username }
})

SQL Injection Protection Methods

1. Parameterized Queries

The most effective defense. Parameters are passed separately from the query string.

const query = 'SELECT * FROM users WHERE id = ?' const results = db.all(query, [userId])
Database drivers: MySQL (?), PostgreSQL ($1, $2), SQLite (? or :name)

2. Input Validation

Validate and sanitize all user input before database operations.

function isValidUsername(username) { // Allow only alphanumeric and underscore return /^[a-zA-Z0-9_]{3,20}$/.test(username); } // Use middleware app.post('/login', validateLogin, (req, res) => { // Safe to process });
Note: Validation is important but not sufficient alone. Always use parameterized queries.

3. Least Privilege

Database accounts should have only necessary permissions.

Application Permissions
Web App SELECT, INSERT, UPDATE on specific tables
Admin Panel DELETE only if absolutely required
Reports SELECT only, no modifications
Never use: Database administrator accounts for web applications!

4. Use ORM Libraries

Modern ORMs provide built-in SQL injection protection.

// Sequelize, TypeORM, Prisma, etc. // All handle parameterization automatically // Example: Prisma const user = await prisma.user.findUnique({ where: { email: email } })
Benefits: Type safety, migration tools, and automatic protection

‚úÖ Best Practices Checklist

  • Always use parameterized queries/prepared statements
  • Validate and sanitize all user input
  • Use stored procedures with parameters when appropriate
  • Implement proper error handling (don't expose SQL errors to users)
  • Use database connection pooling with limited privileges
  • Regularly update database drivers and libraries
  • Implement Web Application Firewall (WAF) rules for SQL injection
  • Conduct regular security audits and penetration testing
  • Use ORM libraries with built-in protection
  • Educate developers about SQL injection risks

Framework-Specific Examples

Express.js with MySQL

const mysql = require('mysql2/promise'); // Create connection pool const pool = mysql.createPool({ host: 'localhost', user: 'app_user', // Limited privileges password: 'secure_password', database: 'app_db', connectionLimit: 10 }); // Safe parameterized query app.get('/user/:id', async (req, res) => { try { const [rows] = await pool.execute( 'SELECT * FROM users WHERE id = ?', [req.params.id] ); res.json(rows); } catch (error) { // Log error but don't expose details console.error('Database error:', error); res.status(500).json({ error: 'Internal error' }); } });

Express.js with PostgreSQL

const { Pool } = require('pg'); const pool = new Pool({ user: 'app_user', host: 'localhost', database: 'app_db', password: 'secure_password', port: 5432, }); app.post('/login', async (req, res) => { const { username, password } = req.body; try { // Safe parameterized query const result = await pool.query( 'SELECT * FROM users WHERE username = $1', [username] ); if (result.rows.length === 0) { res.status(401).json({ error: 'Invalid' }); return; } // Compare hashed password const user = result.rows[0]; const isValid = await bcrypt.compare(password, user.password_hash); if (isValid) { // Create JWT/session res.json({ message: 'Login successful' }); } else { res.status(401).json({ error: 'Invalid' }); } } catch (error) { console.error('Login error:', error); res.status(500).json({ error: 'Internal error' }); } });

Key Framework Features

  • Express.js: Use express-validator for input validation
  • NestJS: Built-in validation with class-validator
  • Fastify: Schema-based validation with Ajv
  • Koa: Middleware-based validation
  • All frameworks: Support parameterized queries with database drivers

Testing for SQL Injection

Test your applications with these inputs:
' -- Single quote " -- Double quote ; -- Semicolon -- -- SQL comment /* */ -- Multi-line comment ' OR '1'='1 -- Always true ' UNION SELECT null -- -- Union attack '; DROP TABLE users; -- -- Destructive '; WAITFOR DELAY '00:00:05' -- -- Time-based ' AND 1=CONVERT(int, @@version) -- -- Error-based

üö® Common Mistakes to Avoid

  • Building dynamic WHERE clauses with string concatenation
  • Using eval() or similar functions with SQL strings
  • Assuming input is safe because it comes from a dropdown or checkbox
  • Using blacklist approaches instead of parameterization
  • Forgetting to escape table or column names (use whitelists instead)
  • Storing passwords in plain text (always hash with bcrypt/argon2)
